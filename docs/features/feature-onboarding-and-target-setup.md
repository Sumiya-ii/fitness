# Onboarding & Target Setup

Last updated: 2026-03-06

## Scope
- Welcome/auth entry for unauthenticated users.
- Multi-step profile/goal onboarding for authenticated users.
- Initial target generation and onboarding completion lock.

## User flow
1. Unauthenticated user lands on Welcome and can go to Sign In/Sign Up.
2. After authentication, mobile fetches `GET /onboarding/status` and uses it as routing truth.
3. If `completed=false`, user is forced through setup stack until profile setup is complete.
4. Mobile submits onboarding payload to `POST /onboarding/complete`.
5. API updates profile and creates first active target in one transaction.
6. App routes user to main experience after completion.

## API surface
- `POST /onboarding/complete`
- `GET /onboarding/status`
- `POST /targets`
- `GET /targets/current`
- `GET /targets/history`

## Data model
- `Profile`: `gender`, `birthDate`, `heightCm`, `weightKg`, `goalWeightKg`, `activityLevel`, `dietPreference`, `onboardingCompletedAt`
- `Target`: `goalType`, macro/calorie targets, `weeklyRateKg`, `effectiveFrom`, `effectiveTo`

## Current logic/rules
- Welcome/auth is the default unauthenticated entry point.
- Onboarding setup routing is driven by backend `GET /onboarding/status` for authenticated users.
- Onboarding is one-time; completed users are rejected on re-submit.
- Target values are generated by rule-based calculator.
- Creating a new target closes previous active target (`effectiveTo=today`).

## Current gaps / next improvements
- No explicit weekly check-in adjustment endpoint yet.
