Request Flow

Mobile App → Firebase Auth Token → NestJS API (AuthGuard) → Prisma → PostgreSQL
API enqueues job → BullMQ (Redis) → Worker processor → AI (OpenAI/Gemini) → Telegram / Push / DB

Authentication Flow

User signs in (email/Google/Apple) → Firebase Auth → ID Token (JWT) → SecureStore on device
API request → Bearer token → Global AuthGuard verifies via Firebase Admin → @CurrentUser() decorator

Subscription & Gating Flow

Purchase (RevenueCat / QPay) → Webhook → API updates Subscription table
Mobile checks → RevenueCat client SDK → Falls back to server /subscriptions/status
Pro features gated by → SubscriptionGuard (API) + useProGate hook (mobile)

Pro-only features: Coach Chat, Photo Analysis

Offline Support

Write operations (meal logs, quick-add, workouts, weight) are queued locally in MMKV when offline. The useSyncQueue hook flushes them in FIFO order when connectivity returns. 4xx errors are dropped; 5xx are retried on next reconnect.

Schema Issues Found

OutboundMessage.id — Missing @id Decorator
The id field has no @id attribute in the Prisma schema. This may cause issues with Prisma client operations (findUnique, etc). The SQL also has no PRIMARY KEY constraint.
Food.verifiedBy — Not a Foreign Key
UUID string field without FK constraint to User. If the admin user is deleted, this reference becomes orphaned.
ModerationQueue — All Soft References
entityId, submittedBy, reviewedBy have no FK constraints. Designed for flexibility but risks orphaned references.

9. Frontend-Backend Mismatches

Backend endpoints NOT called from mobile

POST /targets — No target creation UI
The backend has a full target management system (create, current, history) but the mobile app has zero UI for it. Targets are only created during onboarding. Users cannot adjust targets after onboarding.
POST /body-composition/measurements — No logging UI
Backend can log waist/neck/hip measurements and calculate body fat %, but mobile only reads existing data (from ProgressScreen). No screen for inputting measurements.
POST/PUT/DELETE /foods — No food management UI
Backend supports full food CRUD but mobile only reads foods. May be admin-only, but no admin UI exists either.
GET /weekly-summary — Never called
Dedicated endpoint exists but WeeklySummaryScreen presumably shows data another way or via the chat-injected weekly report.
QPay endpoints — No mobile UI
POST /qpay/invoice, GET /qpay/invoice/:id/status exist but mobile has no QPay payment flow. Mongolian local payment is backend-ready but not surfaced.
GET /notifications/preferences — Not fetched
RemindersScreen updates preferences (PUT) but never fetches current state. Toggles may not reflect actual server state on screen load.
POST /notifications/test — Not used
Test notification endpoint exists but mobile never calls it.
POST /privacy/consent — Not called
Consent tracking infrastructure exists but mobile never records consent events.
GET /privacy/requests — Not fetched
User can trigger data export/deletion but can't check request status.
Admin endpoints — No mobile admin UI
Full moderation queue and message stats endpoints exist but there's no admin screen.

Micronutrients underutilized

Fiber, sugar, sodium, saturated fat — tracked but barely displayed
Backend stores and returns these in meal logs. Photo/voice AI calculates them. But the mobile dashboard and progress screens only prominently show calories, protein, carbs, fat. The detailed micronutrients are fetched but not meaningfully surfaced.

10. Bugs & Issues Found

BUG: OutboundMessage table has no primary key
The id field in the Prisma schema for OutboundMessage is missing the @id decorator. This can cause Prisma client issues when trying to do findUnique, update, or delete operations on outbound messages. The SQL migration also has no PRIMARY KEY constraint. Fix: Add @id to the id field and create a migration.
BUG: RemindersScreen doesn't fetch current notification preferences
When the RemindersScreen loads, it writes preferences (PUT) but never reads them (GET /notifications/preferences). This means the toggle states on the screen may not match the actual server state. If the user set morning reminder OFF on another device, this screen would still show it as ON. Fix: Fetch preferences on screen mount (useFocusEffect).
BUG: Coach message jobId uses Date.now() — potential duplicates
Format is coach-${userId}-${messageType}-${Date.now()}. If the cron runs fast, two jobs for the same user could get the same millisecond timestamp, causing BullMQ to silently deduplicate (one message lost). Fix: Use UUID or ULID for jobId uniqueness.
ISSUE: Food.verifiedBy has no FK constraint
If an admin user is deleted, the verifiedBy reference becomes orphaned. Currently works but could cause data integrity issues over time.
ISSUE: settings.store methods are no-ops
loadUnitSystem() and setUnitSystem() are empty stub functions. The unitSystem is hardcoded to 'metric'. If imperial support is not planned, the store should be cleaned up. If planned, it needs implementation.
ISSUE: ProfileSetupScreen exists but is dead code
File exists at src/screens/onboarding/ProfileSetupScreen.tsx but is not exported from the onboarding index and not registered in any navigator. It should be deleted.
ISSUE: water.store.undoLast() exists but is never called from UI
The undoLast() action in water store exists but no screen calls it. The HomeScreen uses removeCup() instead. Redundant action.
ISSUE: Coach memory block has no size limit
The memory block injected into coach message prompts has no length validation. If summaries are long (500+ characters each × 4 categories), it could consume significant prompt tokens and potentially push the prompt past token limits.
ISSUE: No UI to modify nutrition targets after onboarding
Users complete onboarding → targets are created → but there's no way to change them later. If a user's weight, activity level, or goals change, they're stuck with the original target. The backend has POST /targets but no mobile UI calls it.
ISSUE: Adaptive target notifies but user can't see/confirm adjustment
When the adaptive target processor adjusts calories (too_fast/too_slow), it sends a push/Telegram message. But there's no in-app view showing the user's updated target or letting them accept/reject the change.
NOTE: WeeklySummaryScreen vs weekly-report processor
Backend has a GET /weekly-summary endpoint AND a weekly-report worker processor. The processor injects into chat history. The endpoint exists but mobile doesn't call it directly. May need to decide: use the endpoint for the WeeklySummaryScreen, or keep using chat injection.
NOTE: Telegram Markdown fallback
Coach messages are sent to Telegram with Markdown parsing. If GPT-4o generates invalid Markdown, it falls back to plain text. This works but could be improved by sanitizing AI output before sending.

12. Things Not Wired Together

These are features where the pieces exist across different layers but aren't connected end-to-end.

1. Nutrition Targets — Backend CRUD exists, no mobile UI to edit
Backend: POST /targets (create), GET /targets/current (read), GET /targets/history (list)
Mobile: Targets created during onboarding only. PersonalDetailsScreen lets you edit weight/height/goals but doesn't recalculate or create new targets.
Impact: Users can't adjust calorie/macro targets after onboarding. Weight changes, activity changes, or diet preference changes don't update their targets.
Fix needed: PersonalDetailsScreen should recalculate and POST new targets when profile changes. Add a dedicated "Adjust Targets" screen.
2. Body Composition — Display-only, no way to log measurements
Backend: POST /body-composition/measurements (with Navy body fat formula)
Mobile: ProgressScreen fetches and displays body composition data, but there's no input form to log waist/neck/hip measurements.
Impact: Body composition section shows empty data forever unless someone manually inserts data via API.
Fix needed: Add a "Log Measurement" screen accessible from ProgressScreen.
3. Adaptive Target — Adjusts silently, user can't see current target
Backend: Adaptive target processor calculates adjustment, updates Target in DB, sends notification.
Mobile: No screen shows the current active target. No screen shows target history. User gets a push saying "target adjusted to 1850 kcal" but can't verify it in-app.
Impact: User confusion — they receive notifications about target changes but have no way to see, confirm, or override them.
Fix needed: Show current targets on dashboard or settings. Show target history on progress screen.
4. QPay Mongolian Payments — Complete backend, zero mobile UI
Backend: POST /qpay/invoice (creates QR code + bank deeplinks), GET /qpay/callback (confirms payment), GET /qpay/invoice/:id/status
Mobile: SubscriptionScreen only shows RevenueCat (Apple/Google). No QPay payment option.
Impact: Mongolian users who want to pay via local banks can't. The QPay strategy (bypassing App Store fees) is not accessible.
Fix needed: Add QPay payment option to SubscriptionScreen for Mongolian users.
5. Notification Preferences — Writes but never reads
Backend: GET /notifications/preferences returns current settings, PUT updates them
Mobile: RemindersScreen only calls PUT. Never fetches current state with GET.
Impact: Toggle states on RemindersScreen may not match actual server state. If prefs were changed elsewhere (or set during onboarding), the screen shows default/stale values.
Fix needed: Fetch preferences on screen focus and initialize toggles from server state.
6. Privacy — Delete/export work, but no status tracking
Backend: POST /privacy/export (request), POST /privacy/delete-account (request), GET /privacy/requests (list), POST /privacy/consent
Mobile: Settings triggers delete-account and export but never checks request status or records consent.
Impact: User can't know if their data export is ready or if deletion is processing.
Fix needed: Show request status after triggering. Record health_data consent during onboarding.
7. Weekly Summary — Endpoint exists but mobile uses chat injection
Backend: GET /weekly-summary returns computed data (daysLogged, averages, adherence, weight delta)
Worker: weekly-report processor generates AI narrative and injects into Redis chat history
Mobile: WeeklySummaryScreen exists but likely doesn't call GET /weekly-summary (uses chat history data?)
Impact: Redundant systems. The endpoint returns raw numbers while the worker generates prose. Need to decide which approach to use.
Fix suggestion: Use GET /weekly-summary to populate WeeklySummaryScreen with structured data. Keep worker for Telegram/push delivery.
8. Micronutrients — Tracked end-to-end but invisible to user
Backend: Stores fiber, sugar, sodium, saturated fat in FoodNutrient, MealLogItem snapshots, and MealLog totals
AI: Photo parser and voice parser both extract these micronutrients
Mobile: Dashboard shows only calories, protein, carbs, fat. Extended micronutrients are fetched but barely displayed.
Impact: Users tracking sodium (hypertension) or fiber (digestion) have no visibility into their intake.
Fix suggestion: Add expandable micronutrient section on dashboard or meal detail screen.
9. Coach Messages injected to chat — but chat is Pro only
Worker: Coach messages are injected into Redis chat history for all users
Mobile: CoachChatScreen is gated behind SubscriptionGuard (Pro only)
Impact: Free users receive push/Telegram messages from Coach but can't see their chat history in-app. The chat accumulates messages they can never view.
Fix suggestion: Either make chat read-only for free users (can see coach messages, can't send), or stop injecting into Redis for free users.

13. AI Message Quality & Engagement Problems

Current Issues with Message Quality

Problem: Static reminder messages are repetitive and robotic
Reminders processor uses only 3 variants per type per locale (6 total for morning, 4 for evening). With daily sends, users see the same messages within 2-3 days. Variants like "Өглөөний мэнд! 🌅 — Шинэ өдөр, шинэ боломж" become stale fast.

Fix: Either increase to 10+ variants per type, or switch to AI-generated reminders that incorporate daily context (like coach messages already do). Even simple templates with variable insertion (user name, yesterday's calorie count) would help.
Problem: Meal nudge messages are generic — "Only one meal logged"
Meal nudge processor uses 3 static variants that all say the same thing differently: "you logged one meal, log more." No context about what meal was logged, what time it is, what the user typically eats at this time.

Fix: Use GPT-4o for nudges too, with context: "You logged breakfast (бууз, 2 pieces, 180 kcal). It's 7 PM — what's for dinner?" This leverages the coach persona that's already well-defined.
Problem: Adaptive target messages feel clinical
Adaptive target processor uses static templates that read like system notifications: "Өдрийн зорилт 150 ккал нэмлээ — одоо 1850 ккал." No personality, no explanation of why this matters to the user specifically.

Fix: Use GPT-4o with the Coach persona to explain the adjustment in context: "You've been crushing it — lost 1.2kg in 2 weeks! But let's slow down to protect your muscle. I bumped your target to 1850 kcal. Think of it as one extra bowl of шөл."
Problem: Coach memory summaries are in English, injected into Mongolian prompts
Coach memory processor generates summaries in English ("Alice loves buurz, frequently logs tea"). These are injected into coach message prompts that ask GPT-4o to respond in Mongolian. The language mixing may cause GPT to produce less natural Mongolian.

Fix: Generate memory summaries in the user's locale, or at minimum add a prompt instruction: "The memory block below is in English. Your response must still be entirely in Mongolian."
Problem: No conversation continuity across coach messages
Coach messages are injected into Redis chat history, but subsequent coach messages don't read the previous ones. Each message is generated in isolation. So the morning greeting might say "What's the plan today?" and the evening progress_feedback has no idea what the user said in response (if they replied via chat).

Fix: Include the last 3-5 chat messages as context in the coach message prompt. This creates conversational continuity: "This morning I asked what you'd eat — looks like you went with цуйван. Smart choice, 25g protein!"
Problem: No user feedback loop on AI messages
Users have no way to react to coach messages (like/dislike, helpful/not helpful). This means you can't improve message quality based on user feedback, and you can't tell which message types users value.

Fix: Add simple reaction buttons to coach messages in chat (👍/👎). Store reactions in OutboundMessage.metadata. Use aggregate data to tune message frequency and types.

What's Already Good

Coach persona prompt is excellent
The system prompt for coach messages is detailed, culturally informed, and well-structured. The persona ("brilliant friend who happens to be a nutritionist"), Mongolian food references, and per-message-type tone rules are high quality.
Context injection is thorough
Coach messages include calories, water, protein, streaks, weekly averages, and coach memory. This gives GPT-4o rich data to personalize messages with specific numbers.
Time window validation prevents stale messages
Jobs processed outside their window are silently skipped. Morning greeting at 3 PM? Dropped. This prevents confusing out-of-context messages.
